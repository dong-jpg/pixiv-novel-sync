# AI 创作项目总览单面板设计

日期：2026-07-17
状态：方案 A 已确认，等待规格复核

## 1. 背景

AI 创作页面的“项目总览”当前使用两列四卡片布局，包含“项目信息”“项目概况”“套用蒸馏内容”和“风格控制”。桌面端通过等高拉伸让同一行卡片底部对齐，但四组内容的高度差异较大，因此卡片内部出现明显空白，整体也比“全书规划”页面更零散。

本次采用已确认的方案 A“分区式工作台”：把四张卡片合并为一个完整大面板，内部使用标题、留白和细分隔线建立层级，不再使用嵌套卡片。

## 2. 目标

1. 消除四张等高卡片带来的无效空白和底部不协调。
2. 让用户按“了解当前项目 → 绑定创作基准 → 调整写作风格”的顺序浏览和操作。
3. 延续“全书规划”页面的单容器设计语言，同时保持现有 AI 创作页面的白底、浅边框、轻阴影和品牌色操作样式。
4. 完整保留现有功能、数据绑定、接口调用和独立保存语义。
5. 保证桌面端信息紧凑，移动端内容不溢出且操作顺序自然。

## 3. 范围

### 3.1 本次范围

- 重排 `dashboard_ai.html` 中“项目总览”区域的模板结构和响应式样式类。
- 将四个顶层卡片合并成一个顶层大面板。
- 调整项目信息、项目统计、规划目标、蒸馏内容和风格控制的视觉顺序。
- 更新与项目总览布局直接相关的前端契约测试。
- 对桌面端和移动端执行布局验证。

### 3.2 非目标

- 不修改后端路由、数据库模型或 API 格式。
- 不改变项目资料、蒸馏档案和风格控制的保存接口及保存时机。
- 不修改风格控制注入 Prompt 的逻辑。
- 不调整“全书规划”“章节”“伏笔”“项目状态”和“检索记忆”等其他 `Tab`。
- 不引入新的前端框架、构建工具或通用组件层。
- 不顺带拆分整份 `dashboard_ai.html`；本次重构只处理项目总览的直接结构。

## 4. 信息结构

单一大面板从上到下分为三个一级区域：

1. 作品资料与进度。
2. 蒸馏内容。
3. 风格控制。

三个区域属于同一外层面板，区域之间使用浅色横向分隔线，不再为每个区域添加独立背景、阴影或外层卡片。

### 4.1 作品资料与进度

桌面端使用三部分横向布局：

- 左侧：项目封面。
- 中间：项目名称、项目描述和“保存信息”。
- 右侧：章节、字数、伏笔、待回收四项统计，以及目标字数、预计章节数等规划目标。

内容优先级为：项目身份信息在左，当前完成情况在右。封面保持固定宽高比，不随表单或统计内容变化而拉伸。

保留以下行为：

- 未上传封面时显示稳定渐变和项目名首字符。
- 支持上传、替换和删除封面。
- 图片加载失败时回退到渐变封面。
- 项目名称和描述继续由 `projectMetaForm` 管理。
- “保存信息”继续调用 `saveProjectMeta()`。
- 统计值继续从 `chapters`、`foreshadows` 和 `longformPlan` 实时计算。

封面操作在视觉上靠近封面本身。小屏幕下操作按钮可以换行，但不能遮挡封面、字段或后续内容。

### 4.2 蒸馏内容

蒸馏内容改为紧凑横向设置栏：

- 风格档案选择框。
- 小说档案选择框。
- “管理档案”入口。
- “保存套用”按钮。

说明文字保留，但压缩为标题旁或标题下的一行辅助信息，避免占据过多垂直空间。

保留以下行为：

- 两个选择框继续绑定 `style_profile_id` 和 `novel_profile_id`。
- “保存套用”继续调用 `saveProjectProfiles()`。
- “管理档案”继续跳转 `/dashboard/wizard?tab=distill`。
- 蒸馏内容保存与项目资料、风格控制保存保持独立。

### 4.3 风格控制

风格控制是内容量最大的区域，放在面板底部并获得完整横向空间。

桌面端分为两个内容区：

- 主区域：五个风格滑块，使用两列紧凑网格排列；最后一个滑块允许独占剩余位置，不通过固定高度补齐空白。
- 辅助区域：风格标签、标签输入、额外要求和“保存风格设定”。

保留以下行为：

- 五个滑块继续使用 `STYLE_SLIDERS` 渲染，范围、步进和值绑定均不改变。
- 标签继续支持添加和逐项删除。
- 额外要求继续绑定 `projectMetaForm.style_control.custom`。
- “保存风格设定”继续调用 `saveProjectStyleControl()`。
- 中点 50 的业务含义和当前 Prompt 注入规则不改变。

## 5. 视觉规则

### 5.1 外层面板

- 项目总览只保留一个主要白色面板。
- 圆角、边框和阴影与“全书规划”现有容器保持同一强度。
- 不使用渐变背景，不增加装饰性卡片或嵌套卡片。
- 面板内部各区域使用一致的水平内边距。

### 5.2 层级和间距

- 一级区域标题使用现有紧凑面板标题字号，不使用页面级大标题。
- 辅助说明使用现有 `text-pixiv-gray` 风格。
- 区域之间使用分隔线和稳定内边距，不通过空白占位对齐。
- 统计项可以保留浅灰背景的小型数据块；它们是数据单元，不视为独立页面卡片。
- 主要操作使用品牌色按钮，次要跳转使用描边按钮或文本链接。

### 5.3 响应式

桌面端：

- 作品资料与进度使用“封面、表单、统计”三部分布局。
- 蒸馏内容尽量保持单行，空间不足时允许操作区换行。
- 风格控制使用“滑块主区域、补充设置辅助区域”布局。

移动端：

- 统一切换为单列，顺序保持“封面 → 项目资料 → 统计 → 蒸馏内容 → 滑块 → 标签与额外要求”。
- 封面保持稳定宽高比，不铺满屏幕。
- 输入框、选择框和文本域宽度不超出父容器。
- 操作按钮允许换行，主要保存按钮保持清晰可见。
- 不设置固定内容高度，不使用横向滚动解决布局问题。

## 6. 数据流与保存边界

本次只改变展示结构，现有数据流不变：

1. 打开项目时，现有逻辑填充 `currentProject`、`projectMetaForm`、`chapters`、`foreshadows` 和 `longformPlan`。
2. 项目资料只由 `saveProjectMeta()` 保存。
3. 蒸馏档案绑定只由 `saveProjectProfiles()` 保存。
4. 风格控制只由 `saveProjectStyleControl()` 保存。
5. 封面继续通过现有上传、读取和删除接口处理。

不得为了合并视觉容器而合并三个保存动作，也不得让一个区域的按钮隐式保存另一区域的未提交修改。

## 7. 异常与状态处理

- 封面处理中继续禁用相关按钮并显示处理状态。
- 封面加载失败继续回退到渐变占位，不造成布局跳动。
- 没有规划目标时隐藏目标标签，统计区域仍保持自然高度。
- 没有风格标签时继续显示“未添加标签”。
- 档案列表为空时保留“不套用”选项。
- 保存请求的成功或失败反馈沿用现有全局处理，不在本次新增另一套提示机制。

## 8. 测试设计

### 8.1 模板契约测试

更新现有项目总览布局测试，至少确认：

- 项目总览只有一个顶层 `data-overview-panel`。
- 原四个 `data-overview-card` 不再存在。
- 不再依赖 `items-stretch` 和四个 `h-full` 强制卡片等高。
- 三个一级区域具备稳定、可识别的结构标记。
- `saveProjectMeta`、`saveProjectProfiles`、`saveProjectStyleControl`、封面上传和封面删除入口仍存在。
- 三个保存按钮仍位于各自的内容区域内。

### 8.2 回归测试

- 运行 `tests/test_frontend_library_os.py` 中与 AI 项目总览相关的测试。
- 运行完整测试 `python -m pytest -q`。
- 执行 `git diff --check`，确认模板没有空白错误。

### 8.3 视觉验证

使用浏览器自动化至少检查以下视口：

- 桌面端：1440 × 1000。
- 窄桌面或平板：1024 × 900。
- 移动端：390 × 844。

验收重点：

- 页面只有一个主要总览面板。
- 三个区域边界清楚，没有四卡片式碎片感。
- 面板内部不存在由等高拉伸产生的大块无效空白。
- 封面、字段、统计、滑块和按钮互不遮挡。
- 最长中文标签、按钮文字和辅助说明不溢出容器。
- 移动端没有横向滚动条。

## 9. 验收条件

1. “项目总览”视觉上呈现为一张完整大面板，与“全书规划”的容器语言一致。
2. 内容顺序为“作品资料与进度 → 蒸馏内容 → 风格控制”。
3. 四项统计和规划目标与项目信息处于同一首屏区域。
4. 原有封面、项目资料、蒸馏档案和风格控制功能全部可用。
5. 三组保存动作继续独立，现有后端协议没有变化。
6. 桌面端不再出现四张等高卡片造成的明显空白。
7. 移动端内容顺序自然、无重叠、无溢出、无横向滚动。
8. 定向测试、完整测试和视觉检查全部通过。
