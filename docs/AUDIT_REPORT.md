## 🐛 Bug 检测报告

### 🔴 严重 (Critical) - 需立即修复

#### 1. 信号量泄漏风险 - 可能导致系统死锁
**文件**: `src/pixiv_novel_sync/web/managers.py:314`  
**问题**: `_run_single_task` 在 finally 块外释放信号量，若异常未捕获会导致信号量永不释放，后续所有任务阻塞  
**影响**: 一次异常可能导致整个同步系统永久挂起  
**修复**:
```python
# 修改前
try:
    # ... 任务执行
except Exception:
    pass
self.sync_job_manager._semaphore.release()  # ❌ 在 finally 外

# 修改后
try:
    # ... 任务执行
except Exception:
    pass
finally:
    self.sync_job_manager._semaphore.release()  # ✅ 确保释放
```

#### 2. SQLite 多线程竞态条件
**文件**: `src/pixiv_novel_sync/ai/retrieval.py:61`  
**问题**: SQLite 连接设置 `check_same_thread=False` 后未加锁保护，多线程并发访问会导致 'SQLite objects created in a thread can only be used in that same thread' 错误  
**影响**: AI 检索在高并发时可能崩溃  
**修复**:
```python
# 所有 self.conn 的读写操作都应在锁保护下
with self._lock:
    cursor = self.conn.execute(query, params)
    results = cursor.fetchall()
```

#### 3. 数据一致性问题 - 用户删除顺序错误
**文件**: `src/pixiv_novel_sync/storage/users.py:259`  
**问题**: `delete_user` 先删除 novel_fts/sync_check_list，再删除 novels，若中间失败会导致孤儿记录  
**影响**: 数据库状态不一致，可能导致查询异常  
**修复**: 调整删除顺序，先删从属表，后删主表

### 🟠 高优先级 (High) - 建议本周修复

#### 4. AutoSyncScheduler 锁外读取导致 KeyError
**文件**: `src/pixiv_novel_sync/web/managers.py:219`  
**问题**: `_task_next_run` 字典在锁外读取，但初始化在锁内，可能导致 KeyError  
**修复**: 将整个读取逻辑移入锁保护

#### 5. FTS 索引更新缺少事务保护
**文件**: `src/pixiv_novel_sync/storage/novels.py:371`  
**问题**: `replace_fts` 的 DELETE 和 INSERT 不在同一事务中，INSERT 失败会导致索引丢失  
**修复**: 使用 `transaction()` 上下文包裹

#### 6. 外键约束未启用
**文件**: `src/pixiv_novel_sync/storage/connection.py:19`  
**问题**: SQLite 默认不启用外键约束，即使定义了也不会生效  
**影响**: novels.user_id、novels.series_id 等可能指向不存在的记录  
**修复**:
```python
# 连接初始化时添加
conn.execute("PRAGMA foreign_keys=ON")
```

### 🟡 中等优先级 (Medium) - 建议本月修复

#### 7. requests.Session 未加锁初始化
**文件**: `src/pixiv_novel_sync/storage_files.py:32`  
**问题**: lazy 初始化未加锁，多线程可能创建多个 Session 实例  
**修复**: 双重检查锁模式

#### 8. 资源泄漏 - Session/连接未关闭
**文件**: 多处  
**问题**:
- `storage_files.py:34` - requests.Session 未显式关闭
- `ai/retrieval.py:61` - SQLite 连接未在所有路径关闭
- `ai/providers.py:75` - AIProvider.session 缺少上下文管理器

**修复**: 实现 `__enter__`/`__exit__` 方法，支持 with 语句

#### 9. 过于宽泛的异常捕获
**文件**: `src/pixiv_novel_sync/ai_web.py:128`  
**问题**: 所有 API 端点都用 `except Exception` 捕获，无法区分参数错误(400)和服务器错误(500)  
**修复**: 区分异常类型，返回相应 HTTP 状态码

### 🟢 低优先级 (Low) - 改进建议

#### 10. 安全加固建议
- **SSRF 风险**: `webapp.py:364` 图片代理端点需禁止内网 IP
- **日志泄漏**: 确保不在日志中输出 token/密码
- **CSP 头缺失**: 添加 Content-Security-Policy 防 XSS

## 🚀 优化建议

### 架构优化

#### 1. 存储层重构 (优先级: 高)
**问题**: `storage_db.py` 和 `storage/` 目录职责重叠  
**建议**:
- 废弃 `storage_db.py`，统一迁移到 `storage/` 模块化架构
- 确保所有存储操作走 `storage/connection.py` 的事务管理
- 添加存储层抽象接口，便于未来支持 PostgreSQL

#### 2. Web 应用拆分 (优先级: 中)
**问题**: `webapp.py` (2781行) / `ai_web.py` / `preference_web.py` 三个 Flask 应用  
**建议**:
- 使用 Flask Blueprint 统一注册路由
- 按功能域拆分 (sync/ai/preference/admin)
- 共享中间件 (认证/CSRF/错误处理)

#### 3. 依赖注入改造 (优先级: 中)
**问题**: 全局 manager 对象导致测试困难  
**建议**: 使用依赖注入容器，便于单元测试和模块替换

### 性能优化

#### 1. 数据库查询优化 (优先级: 高)
**问题**: 缺少索引、N+1 查询  
**建议**:
- 为 novels.user_id、novels.series_id 添加索引
- 系列详情页使用 JOIN 而非多次查询
- 添加查询缓存 (Redis/内存)

#### 2. 批量事务改进 (优先级: 高)
**问题**: 单个小说同步独立提交，频繁 commit  
**建议**:
- 批量同步使用单个事务 (50 条/批次)
- 失败时回滚整个批次，记录失败 ID

#### 3. 并发处理优化 (优先级: 中)
**问题**: 信号量硬编码，无法动态调整  
**建议**: 并发数改为配置项，根据系统资源动态调整

### 代码质量改进

#### 1. 类型注解完整性 (优先级: 中)
**现状**: 部分函数缺少类型注解  
**建议**: 使用 `mypy` 强制类型检查，提升代码可维护性

#### 2. 代码异味修复 (优先级: 中)
**发现**:
- 长函数 (>100行): `sync_engine.py` 多处
- 高复杂度函数 (圈复杂度>10): 建议拆分

**建议**: 使用 `pylint`/`flake8` 检查，逐步重构

#### 3. 测试覆盖度提升 (优先级: 高)
**现状**: 仅部分模块有测试  
**建议**:
- 核心同步逻辑单元测试 (目标 80% 覆盖)
- AI 功能集成测试
- Web API 端到端测试

### 用户体验优化

#### 1. 移动端适配 (优先级: 高)
**问题**: 操作栏在小屏幕隐藏，用户无法操作  
**建议**: 添加底部浮动操作栏或汉堡菜单

#### 2. 实时反馈改进 (优先级: 高)
**问题**: 缺少 Toast 提示，依赖 console  
**建议**: 引入 Toast 组件，统一成功/警告/错误提示

#### 3. 首次使用引导 (优先级: 高)
**问题**: 新用户不知道从哪开始  
**建议**: 首次登录显示配置向导 (Token → 同步设置 → 开始同步)

#### 4. 错误消息友好化 (优先级: 高)
**问题**: 技术术语多，缺少操作建议  
**建议**:
- 简化错误消息 (避免 CSRF/token 等术语)
- 提供解决建议 ("请刷新页面重试")
- 按严重程度分级 (INFO/WARN/ERROR/FATAL)

#### 5. API 一致性改进 (优先级: 高)
**问题**: 错误响应格式不统一  
**建议**: 统一为 `{success: bool, data/error: {...}}`
